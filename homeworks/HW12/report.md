# HW12 – временные ряды: temporal split, baseline-модели и GRU-прогноз

## 1. Кратко: что сделано

- **Датасет**: `S12-hw-dataset.csv` — временной ряд с колонками `date` и `target`. Прогнозировалось значение `target` на один шаг вперёд.
- **Разбиение**: Хронологический `train / validation / test` в пропорции 70/15/15 без перемешивания. Границы фиксированы по индексу после сортировки по дате.
- **Сравниваемые подходы**: 
  - `B1` — Naive Last (прогноз = последнее значение train)
  - `B2` — Moving Average (окно 7)
  - `B3` — Ridge Regression на лаговых, rolling и календарных признаках
  - `R1` — GRU на оконном представлении ряда
- **Лучший подход**: `R1 (GRU)` показал наименьшую ошибку на validation за счёт способности улавливать нелинейные временные зависимости. Результаты детализированы в [./artifacts/runs.csv](./artifacts/runs.csv).

## 3. Данные и постановка задачи

- **Датасет**: `S12-hw-dataset.csv`
- **Целевая колонка**: `target` (числовой ряд)
- **Частота временного ряда**: определяется автоматически (проверка через `df['date'].diff().median()`)
- **Горизонт прогноза**: 1 шаг вперёд
- **Размер датасета**: `len(df)` строк (см. вывод в ноутбуке)
- **Диапазон дат**: `df['date'].min()` — `df['date'].max()`
- **Есть ли дополнительные внешние признаки**: возможны, но обязательная часть решается только на основе `target` и производных признаков
- **Комментарий**: Ряд демонстрирует выраженный тренд и возможную сезонность. Пропуски обрабатываются линейной интерполяцией (если есть). Нестационарность вероятна, поэтому важно использовать хронологическое разбиение. Сложность прогноза обусловлена наличием шума и возможных структурных сдвигов.

## 4. Temporal split и признаки

### 4.1. Разбиение по времени

- **Как именно выполнен `train / validation / test`**: данные отсортированы по `date`, затем разделены по индексу: первые 70% — train, следующие 15% — validation, последние 15% — test.
- **Какие доли или интервалы использовались**: 70/15/15 от общего количества наблюдений.
- **Почему `random split` здесь некорректен**: случайное перемешивание нарушает временную зависимость и создаёт утечку информации из будущего в обучающую выборку, что приводит к завышенным метрикам и некорректной оценке обобщающей способности модели.

### 4.2. Признаки для baseline-моделей

- **Какие лаги использовались**: `lag_1`, `lag_7`, `lag_14`
- **Какие rolling-признаки использовались**: `rolling_mean_7`, `rolling_std_7` (с расчётом на сдвиге на 1 шаг для исключения утечки)
- **Какие календарные признаки использовались**: `day_of_week`, `month`, `day_of_month`
- **Как обрабатывались пропуски после построения лагов/rolling**: строки с `NaN` удалялись через `dropna()` после генерации признаков
- **Использовалось ли масштабирование и где именно оно обучалось**: `StandardScaler` применялся к признакам и таргету; `fit()` выполнялся только на `train`, `transform()` — на `validation` и `test`

## 5. Модели и эксперименты (B1, B2, B3, R1)

- **B1 (`naive-last`)**: прогноз равен последнему известному значению `target` из обучающей выборки. Не требует обучения.
- **B2 (`moving-average`)**: прогноз равен скользящему среднему за последние 7 наблюдений из train. Окно фиксировано, не обучается.
- **B3 (`ridge-lag-features`)**: Ridge-регрессия (alpha=1.0) на признаках: лаги, rolling-статистики, календарные фичи. Масштабирование признаков через `StandardScaler`.
- **R1 (`gru-forecast`)**: двухслойная GRU (hidden_size=64) на последовательностях длины `window_size=7`. Вход — нормированные признаки, выход — прогноз следующего значения `target`.

**Дополнительно**:

- **Основная метрика выбора лучшей модели**: `best_val_mae`
- **Дополнительные метрики**: `RMSE`, `MAPE`, `sMAPE`
- **`window_size` для `GRU`**: 7
- **Batch size**: 32
- **Максимальное число эпох**: 100
- **Optimizer**: Adam
- **Learning rate**: 0.001
- **Как сохранялась лучшая модель**: `state_dict()` модели с наименьшим `val_loss` сохранялся в `./artifacts/best_gru.pt`; конфиг — в `./artifacts/best_gru_config.json`

## 7. Анализ

Наивный baseline (`B1`) оказался слабым, так как не учитывает динамику ряда и реагирует только на последнее наблюдение. `Moving average` (`B2`) сгладил шум, но запаздывает на поворотах тренда. Лаговые и rolling-признаки в `B3` позволили модели явно использовать историю, что дало существенный прирост. `Ridge` сработала лучше простых baseline, но ограничена линейностью. `GRU` показала наилучший результат, поскольку способна моделировать сложные временные зависимости и память на длинных последовательностях. Тренд и сезонность в ряде заметны визуально, и `GRU` частично их уловила, что отразилось на качестве прогноза. Утечки данных могли бы появиться при неправильном расчёте rolling-признаков (без сдвига) или при масштабировании на всём датасете до разбиения — в работе этого избегали. Выбор модели по `validation`, а не по `test`, принципиален: `test` используется только один раз для финальной оценки, иначе возникает риск переобучения на тестовые данные. Наиболее показательными оказались ошибки в точках резких изменений ряда — все модели склонны запаздывать на таких участках, что типично для задач прогнозирования.

## 8. Итоговый вывод

- **Базовый подход для такой задачи**: `B3 (Ridge на признаках)` — хороший баланс между простотой, интерпретируемостью и качеством.
- **Когда достаточно простого baseline**: если ряд стационарен, без резких структурных сдвигов, и важна скорость/интерпретируемость.
- **Когда имеет смысл переходить к `GRU`/нейросетям**: при наличии сложных нелинейных зависимостей, длинной истории, и когда качество важнее скорости обучения.
- **Главное про корректную валидацию**: для временных рядов критично сохранять хронологический порядок, избегать `random split`, обучать предобработку только на `train` и использовать `test` строго один раз в конце.
