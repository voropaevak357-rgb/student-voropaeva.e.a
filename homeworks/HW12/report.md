# HW12 – временные ряды: temporal split, baseline-модели и GRU-прогноз

## 1. Кратко: что сделано

- **Датасет**: `S12-hw-dataset.csv` — почасовой временной ряд с целевой переменной `target`.
- **Прогнозировалось**: значение `target` на один шаг вперёд (горизонт = 1 час).
- **Temporal split**: данные разделены хронологически: 70% train (2025-01-01 — 2025-05-06), 15% val (2025-05-07 — 2025-06-02), 15% test (2025-06-03 — 2025-06-29).
- **Сравниваемые подходы**:
  - `B1`: Naive-last (прогноз = последнее известное значение)
  - `B2`: Moving Average (скользящее среднее за окно)
  - `B3`: Ridge-регрессия на лаговых и rolling-признаках
  - `R1`: GRU-сеть для прогнозирования временных рядов
- **Лучший подход**: `B1` (Naive-last) показал наименьшую MAE на валидации. Это объясняется высокой автокорреляцией ряда: следующее значение сильно зависит от предыдущего, поэтому простая стратегия «последнее значение» оказывается эффективной.

## 3. Данные и постановка задачи

- **Датасет**: `S12-hw-dataset.csv`
- **Целевая колонка**: `target`
- **Частота временного ряда**: 1 час (`0 days 01:00:00`)
- **Горизонт прогноза**: 1 шаг (1 час)
- **Размер датасета**: 4320 наблюдений
- **Диапазон дат**: 2025-01-01 00:00:00 — 2025-06-29 23:00:00
- **Есть ли дополнительные внешние признаки**: нет, только временная метка `date` и `target`
- **Комментарий**: Ряд демонстрирует выраженную краткосрочную автокорреляцию и слабую сезонность. Тренд умеренный, выбросы отсутствуют, пропусков нет. Высокая предсказуемость на 1 шаг делает задачу подходящей для проверки как простых, так и сложных моделей.

## 4. Temporal split и признаки

### 4.1. Разбиение по времени

- **Как выполнен split**: хронологическое разбиение через `pd.to_datetime()` + `sort_values()` + индексация по позициям.
- **Доли**: 70% train (3024 obs), 15% val (648 obs), 15% test (648 obs).
- **Почему random split некорректен**: при случайном перемешивании нарушается временная структура, модель может «увидеть будущее» при обучении, что приводит к завышенным метрикам и невалидным выводам.

### 4.2. Признаки для baseline-моделей

- **Лаги**: `[1, 2, 3, 6, 12, 24]` часов
- **Rolling-признаки**: скользящие средние и стандартные отклонения за окна `[6, 12, 24]`
- **Календарные признаки**: `hour`, `dayofweek`, `day`, `month`, циклическое кодирование (`sin/cos`) для `hour` и `dayofweek`
- **Обработка пропусков**: `dropna()` после создания лагов и rolling-статистик
- **Масштабирование**: `StandardScaler`, обученный **только на train**, затем применён к val/test (для признаков и для target отдельно)

## 5. Модели и эксперименты (B1, B2, B3, R1)

- **B1 (`naive-last`)**: прогноз = последнее известное значение `target` в окне.
- **B2 (`moving-average`)**: прогноз = среднее значение `target` за последние `window_size` наблюдений.
- **B3 (`ridge-lag-features`)**: Ridge-регрессия на признаках: лаги, rolling-статистики, календарные признаки.
- **R1 (`gru-forecast`)**: однослойная GRU с `hidden_size=64`, `num_layers=2`, `dropout=0.2`, вход — нормированный ряд длины `window_size=30`.

**Дополнительно**:
- **Основная метрика**: `MAE` (Mean Absolute Error)
- **Дополнительные метрики**: `RMSE`, `MAPE`
- **`window_size` для GRU**: 30
- **Batch size**: 32
- **Максимальное число эпох**: 50
- **Optimizer**: Adam, `lr=1e-3`
- **Сохранение лучшей модели**: по минимальному `val_loss`, состояние сохраняется в `best_gru.pt`, конфиг — в `best_gru_config.json`

## 7. Анализ

Наивный baseline (`B1`) оказался сильным из-за высокой автокорреляции ряда: следующее значение тесно связано с предыдущим, поэтому стратегия «последнее значение» минимизирует ошибку на 1 шаг. `Moving average` (`B2`) сглаживает шум, но запаздывает при изменениях, что ухудшает точность. Лаговые признаки в `B3` позволили модели учитывать более длинную историю, а rolling-статистики — адаптироваться к локальной волатильности, однако линейная модель не всегда улавливает нелинейные зависимости. `GRU` (`R1`) теоретически способна моделировать сложные временные зависимости, но на данном ряде с коротким горизонтом и сильной автокорреляцией её преимущество не проявилось. Тренд и сезонность в данных выражены слабо, поэтому сложные архитектуры не дают значимого выигрыша. Утечки данных могли возникнуть при некорректном создании лагов (если бы использовались будущие значения) или при масштабировании на всём датасете вместо только train. Выбор модели по `validation`, а не по `test`, принципиален: `test` используется только для финальной оценки, чтобы избежать переобучения на тестовых данных. Наиболее показательными оказались ошибки в точках резких изменений ряда, где все модели демонстрировали запаздывание.

## 8. Итоговый вывод

- **Базовый подход**: `Naive-last` или `Ridge` на лаговых признаках — просты, интерпретируемы и эффективны при высокой автокорреляции.
- **Когда достаточно baseline**: при коротком горизонте прогноза, сильной автокорреляции, ограниченных данных или требованиях к интерпретируемости.
- **Когда переходить к GRU/нейросетям**: при длинном горизонте, сложных нелинейных зависимостях, наличии множества внешних признаков и больших объёмах данных.
- **Главный урок по валидации**: для временных рядов критически важно сохранять хронологический порядок при разбиении и никогда не использовать будущие данные при создании признаков или масштабировании.
