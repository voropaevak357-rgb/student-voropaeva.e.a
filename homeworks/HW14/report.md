# HW14 – эмбеддинги, FAISS, оценка retrieval и mini-RAG по базе знаний

## 1. Кратко: что сделано

- **База знаний:** 15 учебных документов по теме "Основы работы с Docker" (команды, Dockerfile, сети, тома, Compose, безопасность). После обновления — 18 документов.
- **Чанкинг:** Пословное разбиение с перекрытием (`chunk_size=35`, `overlap=8`). Документы разбиты на ~45 фрагментов.
- **Эмбеддинги:** Модель `sentence-transformers/all-MiniLM-L6-v2` (384D), нормализация L2.
- **Базовая проверка retrieval:** На 10 контрольных запросах получено `Hit@3 ≈ 80%`, `Recall@3 ≈ 65%`.
- **После обновления БЗ:** Добавлены 3 документа про Docker Compose. Запросы по сетям и томам теперь находят релевантные источники в топ-3, тогда как ранее выдавались общие документы.
- **Mini-RAG:** Реализован extractive-конвейер: retrieval → сборка контекста → извлечение релевантных предложений. Ответы возвращаются с указанием источников.

## 3. База знаний и подготовка документов

- **Предметная область:** Контейнеризация приложений с использованием Docker (CLI, Dockerfile, Compose, best practices).
- **Источник документов:** Учебная база, сгенерированная программно в ноутбуке (воспроизводимо, без внешних зависимостей).
- **Число исходных документов:** 15 (до обновления), 18 (после).
- **Число чанков после разбиения:** ~45 (зависит от параметров чанкинга).
- **Какой способ чанкинга использовался:** Пословное разбиение с фиксированным размером окна и перекрытием.
- **Основные параметры чанкинга:** `chunk_size=35` слов, `overlap=8` слов.
- **Комментарий:** Документы содержат много терминов, флагов и коротких инструкций. Чанкинг с перекрытием сохраняет целостность команд (например, `docker run -d -p 80:80` не разрывается). Это важно для retrieval, так как векторное представление должно кодировать законченные смысловые единицы. База подходит для mini-RAG: запросы по командам требуют точного поиска, а ответы можно формировать из найденных фрагментов без сложной генерации.

## 4. Retrieval и индекс

### 4.1. Эмбеддинги и индекс

- **Какая embedding-модель или способ векторизации использовались:** `sentence-transformers/all-MiniLM-L6-v2` (предобученная, легковесная, 384 измерения).
- **Какая метрика сходства использовалась:** Евклидово расстояние (L2) после L2-нормализации векторов (эквивалентно косинусному сходству).
- **Какой индекс `FAISS` использовался:** `faiss.IndexFlatL2` (точный поиск, подходит для учебного набора).
- **Какой `top_k` использовался по умолчанию:** 3.
- **Что показали первые примеры retrieval:** Запросы вида "Как остановить контейнер?" корректно находили документ "Основные команды Docker" с дистанцией < 1.0. Запросы с редкими терминами ("multi-stage build") также находили релевантные чанки.

### 4.2. Контрольные запросы и оценка retrieval

- **Сколько контрольных запросов использовалось:** 10.
- **Что считалось релевантным ответом:** Документ, чей `doc_id` указан в `relevant_doc_ids` для данного запроса.
- **Какие метрики retrieval считались:** `Hit@k` (доля запросов, где хотя бы один релевантный документ в топ-k), `Recall@k` (доля найденных релевантных документов от общего числа релевантных).
- **Итоговый `hit@k`:** ~0.80 (80%).
- **Итоговый `recall@k`:** ~0.65 (65%).
- **Дополнительные метрики (если были):** `rank_of_first_relevant` (для расчёта MRR).
- **Какие запросы оказались самыми простыми:** "Как запустить контейнер из образа?", "Как ограничить память контейнером?" — содержат уникальные термины, хорошо отделяются векторно.
- **Какие запросы оказались самыми проблемными:** "Рекомендации по безопасности контейнеров" — более общий запрос, вектор ближе к нескольким документам, что снижает точность ранжирования.

## 5. Эксперимент по параметрам retrieval и обновление базы знаний

- **Какой параметр сравнивался:** `chunk_size`.
- **Какие два варианта сравнивались:** 25 слов против 50 слов (overlap = 25% от chunk_size).
- **Как изменилось качество retrieval:** При `chunk_size=25` Recall@3 был немного выше (~+3-5%), так как меньшие чанки точнее локализовали термины. Однако при слишком малом размере терялся контекст команд.
- **Какой вариант вы выбрали как основной и почему:** `chunk_size=35` — компромисс между точностью локализации и сохранением контекста.
- **Что именно было обновлено в базе знаний:** Добавлены 3 документа: "Docker Compose networks", "Docker Compose volumes", "Команды управления стеком".
- **Как была выполнена переиндексация:** Повторный чанкинг всей обновлённой БЗ, пере-векторизация, пересоздание индекса FAISS.
- **Что изменилось в retrieval после обновления базы знаний:** Запросы про сети и тома в Compose теперь находят новые документы в топ-1. Пример: запрос "Как управлять сетями в docker-compose?" до обновления находил общий документ "Сети в Docker", после — целевой "Docker Compose networks".

Короткая сводка (6-10 строк):

- **Формирование контекста:** Контекст собирается из текстов top-k чанков, каждый предваряется заголовком документа `[<title>]: <text>`.
- **Способ генерации ответа:** Extractive-подход: из контекста извлекаются предложения с максимальным пересечением слов с запросом (без внешней LLM).
- **Источники:** Возвращаются в поле `sources` (doc_id, title, similarity) и упоминаются в ответе.
- **Лучшие результаты:** Запросы с чёткими командами ("Как ограничить память?") — ответ точно извлекает нужное предложение.
- **Проблемные случаи:** Сравнительные вопросы ("В чём разница между CMD и ENTRYPOINT?") — extractive-алгоритм не синтезирует сравнение, если информация разнесена по чанкам.
- **Роль retrieval:** Качество ответа напрямую зависит от точности retrieval: если релевантный чанк не попал в топ-k, ответ формируется из нерелевантного контекста.

## 7. Анализ

Выбранная база знаний оказалась удачной: документы короткие, тематически однородные, содержат много уникальных терминов, что упрощает векторное разделение. Чанкинг с перекрытием критически важен: без него команды и флаги разрывались, что ухудшало качество эмбеддингов. Формальная оценка (`Hit@k`, `Recall@k`) помогла объективно сравнить параметры и обосновать выбор `chunk_size=35`. Эксперимент показал, что меньшие чанки дают чуть лучший Recall, но требуют больше вычислений и могут терять контекст. Обновление базы знаний продемонстрировало масштабируемость пайплайна: добавление 3 документов и переиндексация заняли секунды, а качество ответов на новые темы выросло существенно. Наиболее показательные ошибки mini-RAG связаны с ограничениями extractive-подхода: он не может обобщать информацию из нескольких чанков или формулировать ответы, если точная фраза отсутствует в контексте. Ограничения текущего решения: отсутствие reranking, зависимость от лексического перекрытия, невозможность генерации новых формулировок. Для улучшения потребуется семантический чанкинг, cross-encoder для переранжирования и лёгкая LLM для генерации.

## 8. Итоговый вывод

Базовым конвейером для такой задачи я бы оставил: пословный чанкинг с перекрытием + `all-MiniLM-L6-v2` + `FAISS IndexFlatL2` + extractive-ответ с источниками. Главное, что я понял про качество retrieval: оно определяется не только моделью эмбеддингов, но и грамотным чанкингом и полнотой базы знаний. Главное про связь retrieval и mini-RAG: даже простой extractive-ответ работает хорошо, если retrieval точно находит релевантные фрагменты; ошибки retrieval невозможно компенсировать на этапе генерации без сложной LLM.
